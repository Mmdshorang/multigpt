# Incident Report: Auth Corruption via Account Switch

**Date:** 2026-05-05  
**Affected:** MultiCodex macOS app v0.4.0+  
**Severity:** High — data corruption (auth tokens)  
**Status:** Fixed in `main` (commits `2a7fe74`, `a4b7ff5`)

---

## 1. Summary

Three separate Codex accounts (`moein-2`, `personal-mh`, `vawece@blues.my.id`) were corrupted and ended up sharing the **identical OAuth auth token**. This caused:

- All three accounts showing the same email (`mhmoazennia@gmail.com`) and no usage data (`-`)
- `codex-cli` failing with: *"Your access token could not be refreshed because your refresh token was already used"*
- OAuth refresh tokens being consumed by one account and invalidated for the others (single-use tokens)

The root cause was `AuthSwapService.switchToAccount` blindly writing whatever was in `~/.codex/auth.json` back into the previous account's storage during a switch — without verifying that the system auth actually belonged to that account.

---

## 2. Root Cause Analysis

### 2.1 How Account Switching Works (Legacy Path)

MultiCodex uses an **auth-swap pattern** for legacy accounts:

1. Copy target account's auth from `~/.config/multicodex/accounts/<name>/auth.json` → `~/.codex/auth.json`
2. Run `codex` CLI commands (which reads `~/.codex/auth.json`)
3. After commands finish, copy `~/.codex/auth.json` back → `~/.config/multicodex/accounts/<name>/auth.json`

This is necessary because `codex-cli` only looks at `~/.codex/auth.json`.

### 2.2 The Bug

```swift
// AuthSwapService.swift (before fix)
if let previousName = previousAccountName,
   let currentSystemAuth = try? Data(contentsOf: URL(fileURLWithPath: paths.defaultCodexAuthPath))
{
    try writeAuthData(currentSystemAuth, account: previousName, paths: paths)
}
```

**Problem:** This preserves whatever is currently in `~/.codex/auth.json` into the previous account's slot. If the system auth was modified externally (e.g. via `codex login` in Terminal, or a previous failed restore left different auth in place), the **wrong token gets permanently saved** to the previous account.

### 2.3 How Corruption Spread

| Step | Action | System Auth (`~/.codex/auth.json`) | Account A (`moein-2`) | Account B (`personal-mh`) |
|------|--------|-----------------------------------|----------------------|--------------------------|
| 1 | Switch A → B | B's token | **B's token** (wrong!) | B's token |
| 2 | User runs `codex login` in Terminal | C's token | B's token | B's token |
| 3 | Switch B → A | A's token | A's token | **C's token** (wrong!) |
| 4 | Now B and C share a token | — | — | C's token = B's old token |

After this, all three accounts have cloned tokens. When MultiCodex refreshes Account A, the refresh token is consumed. Accounts B and C now have stale tokens → API calls fail → usage shows `-`.

### 2.4 Why Refresh Tokens Are Single-Use

OpenAI's OAuth implementation issues a **new refresh token** on every refresh. The old one is invalidated immediately. When three accounts share the same token:

1. Account A refreshes → gets new token → writes it back
2. Account B tries to refresh with old token → **"refresh token already used"**
3. Account C tries to refresh → same error

This cascades into complete auth failure for all cloned accounts.

## 3. Issues Discovered During Investigation

### Issue 1: Auth Swap Blind Preservation (CRITICAL)
**File:** `Sources/MultiCodex/Infrastructure/Codex/Accounts/AuthSwapService.swift`  
**Lines:** 20-25  
**Impact:** Corrupts previous account's auth with wrong token  
**Fix:** Compare identities before preserving. If mismatch, throw `externalAuthDetected`.

### Issue 2: Missing Identity Verification During Refresh
**File:** `Sources/MultiCodex/Infrastructure/Codex/Usage/RateLimitsFetcher.swift` (serial path)  
**Lines:** 191, 296  
**Impact:** `snapshotDefaultAuthToAccount` saves whatever is in `~/.codex/auth.json` after RPC/API calls  
**Note:** This is less dangerous than Issue 1 because `withAccountAuth` restores previous auth on error, but still vulnerable if `codex-cli` internally refreshes the token.

### Issue 3: `timedOut` Sendable Mutation Warning
**File:** `Sources/MultiCodex/Infrastructure/Codex/Usage/RateLimitsFetcher.swift`  
**Lines:** 181-204  
**Impact:** Compiler warning — potential race condition in parallel fetch  
**Fix:** Wrapped `timedOut` in `TimedOutBox` class.

### Issue 4: Unused Variable `failed`
**File:** `Sources/MultiCodex/Features/Shared/AccountExportService.swift`

### Commit `a4b7ff5` — Auth Corruption Prevention

#### 4.1 Identity Comparison in `AuthSwapService`

```swift
// AuthSwapService.swift
private static func authDataMatches(_ lhs: Data, _ rhs: Data) -> Bool {
    guard let leftPayload = try? JSONSerialization.jsonObject(with: lhs) as? [String: Any],
          let rightPayload = try? JSONSerialization.jsonObject(with: rhs) as? [String: Any]
    else { return lhs == rhs }
    let leftID = (leftPayload["tokens"] as? [String: Any])?["account_id"] as? String
    let rightID = (rightPayload["tokens"] as? [String: Any])?["account_id"] as? String
    if let leftID, let rightID { return leftID == rightID }
    return lhs == rhs
}
```

Before preserving displaced auth, the system auth is compared to the previous account's stored auth. If the `account_id` differs, switch is blocked.

#### 4.2 New Error Type

```swift
enum AuthSwapError: LocalizedError {
    case externalAuthDetected(
        previousAccount: String,
        previousIdentity: ResolvedAccountIdentity?,
        systemIdentity: ResolvedAccountIdentity?
    )
}
```

#### 4.3 Force Switch Option

Added `forceSwitchAccount` method that bypasses the guard:

```swift
func forceSwitchAccount(name: String) async throws {
    _ = try forceSwitchAccountNow(name: name)
    await CodexRPCSession.shared.shutdown()
}
```

This uses `AuthSwapService.switchToAccount(..., force: true)` which skips identity verification.

#### 4.4 UI Alert with Dual Actions

When switch is blocked, the menu bar shows a combined alert:

- **"Import Account"** — imports the external auth as a new MultiCodex account
- **"Force Switch"** — proceeds with switch (user accepts risk)

`MenuAlertState` now supports `secondaryActionTitle` and `secondaryAction`. `AlertActionCard` renders a second pill button when present.

#### 4.5 Protocol & Mock Updates

- Added `forceSwitchAccount(name:)` to `CodexAccountServicing` protocol
- Added implementation to `CodexAccountService` and mock in tests

## 7. Future Work / Recommendations

1. **Migrate to managed homes fully** — The auth-swap pattern is fundamentally fragile. Managed homes (`CODEX_HOME` per account) eliminate the need for swapping entirely. The migration infrastructure exists but is not enforced.

2. **Add token fingerprint logging** — Log a hash of `account_id` + `refresh_token` prefix on every refresh to detect cloning early.

3. **Periodic integrity check** — During refresh, verify that all accounts have unique `account_id` values. Flag duplicates as corrupted.

4. **Reconciliation on switch** — Before switching, verify the system auth identity matches the expected current account. If not, trigger reconciliation instead of switching.

---

## 8. Recovery for Affected Users

If your accounts were corrupted (same email, `-` usage):

1. **Delete the corrupted accounts** from MultiCodex Settings → Accounts
2. **Re-login each account separately** using "Open Login in Terminal"
3. **Do not import** the same `~/.codex/auth.json` into multiple accounts

To prevent future issues, ensure each account has a unique login session.
